# 자동화 스코프 원칙

자동화 설계 시 **결정론적 룰베이스만 자동 사이드카**, **LLM·판단 영역은 명시 호출 도구**로 분리. 자체 구축 전에 공식 도구 존재 여부 먼저 확인 (YAGNI). 메모리 룰 위반은 발견 즉시 시정.

> 사용자 명시 (2026-05-07): 자체 review-auditor 에이전트 설계 진행 중 OpenAI `codex-plugin-cc` 공식 제공 발견 → YAGNI 적용해 자체 구축 폐기.

## 1. 자동 사이드카 vs 명시 호출 분리 (MUST)

### 자동 사이드카 (daemon thread, cron) 에 둘 수 있는 것
- **결정론적 룰베이스만** — `os.waitpid` 호출, filesystem stat, regex/path 비교, git CLI 호출
- 판정 결과가 입력에 의해 100% 결정 + 외부 비용 0
- 예시: T-403 좀비 GC, T-413 T1 정형 (파일 실재 + status.json), T-413 T2 구조 (target ↔ git diff 매칭)

### 자동 사이드카에 두지 말 것
- **LLM 의미 검증, 판단 영역, 외부 API** — 비용·오탐·비결정성
- 예시: 회피한 T-413 Phase 3 (T3 LLM 의미 검증)
- 자동 호출 시 폭발: 60초 폴링 × 다수 티켓 × 매 호출 토큰 비용

### 명시 호출 도구로 분리
- 사용자가 의식적으로 발사 → 비용 통제 + 의도 명확
- 진입점: 슬래시 명령 (`/codex:adversarial-review`) 또는 CLI wrapper
- 결과: 사용자 화면에 직접 노출 (별도 사이드카 결과 파일과 분리 권장)

## 2. 공식 도구 우선 — YAGNI (MUST)

자체 에이전트/스킬을 구축하기 전에 다음 순서로 확인:

1. **공식 plugin/integration** — OpenAI, Anthropic, GitHub 등이 동등 기능 제공 중인지 web 검색
2. **CLI 통합** — codex CLI, gh CLI 같은 외부 도구가 subprocess 로 호출 가능한지
3. **기존 `.claude/skills/` 의 review-* / audit-* 스킬 조합** 으로 가능한지
4. 그 다음에야 자체 구축 검토

### 자체 구축의 비용
단지 코드 라인 수가 아님:
- 유지보수 부담
- 기존 도구의 진화에서 뒤처짐
- 같은 기능의 양립 (혼동·디버깅 비용)

### 사례
2026-05-07 자체 review-auditor + my-audit-ticket-review 스킬 + audit-ticket 슬래시 명령 약 350줄 설계 진행 중, 사용자가 `/codex:review` `/codex:adversarial-review` (openai/codex-plugin-cc) 존재 지적 → 본 자체 구축 즉시 폐기.

## 3. 비용·빈도 의식적 통제 (MUST)

폴링 주기를 정할 때 "기본 60초" 대신 다음을 평가:
- 실제 사건 발생 빈도 — 분 단위? 시간 단위? 일 단위?
- 호출당 비용 — filesystem stat (~0.1ms) vs LLM API (수만 토큰)
- 미스 latency 허용 범위 — review 진입 후 첫 검증까지 1분/5분/10분 어느 게 충분한가

### 권고 default

| 작업 유형 | 권고 주기 |
|-----------|----------|
| filesystem/git CLI 같은 무비용 작업 | 60초 OK (T-403 GC 좀비 reap) |
| review/build/PR 같은 사람 주도 사건 | 5분 OK (UX 영향 미미) |
| LLM 호출 포함 | 자동 X (명시 호출만) |

### 대안 패턴
**이벤트 기반 트리거** — Board 의 FileWatcher 같은 기존 기제로 변경 감지 시 1회 발사 + 폴링은 안전망 (장주기 5~10분).

## 4. 메모리 룰 위반 즉시 시정 (MUST)

룰 위반 발견 시 사용자가 즉각 짚음 → 같은 세션에서 정정 처리. 보류·박제로 미루지 말 것.

증거: 2026-05-07 자동 채번 회귀로 T-904/T-905 부여됨 (메모리 룰 "디버그 영역 폐지 후 신규는 일반 채번" 위반). 사용자 지적 즉시 T-904 → T-415, T-905 → T-416 수동 정정 + 채번 로직 자체 회귀를 T-417 로 별도 박제.

룰 위반 패턴은 단순 시정만 하지 말고 **로직 자체 회귀로 박제** — 같은 위반의 재발 방지.

## 5. 회피 사항 (사례 박제)

- 사이드카에서 LLM 자동 호출 (비용 + 오탐 + 비결정)
- 자체 review-auditor 에이전트 구축 (Codex Plugin 공식 동등 이상)
- T2 디렉터리 매칭에서 `startswith` 만 사용 (`.claude-organic/` prefix 충돌 → substring 매칭으로 회피)
- 신규 메모리 파일 남발 (사용자 지시: 중복이면 보강, 아니면 신규)

## 적용 우선순위 (설계 결정 시 자문)

다음 순서로 자문:

1. **결정론적인가?** (룰/regex/CLI/filesystem) → 사이드카 OK
2. **LLM 또는 외부 비용 발생?** → 명시 호출 도구로
3. **공식 도구 존재?** → 자체 구축 회피, 통합만
4. **폴링 주기·호출 빈도가 사건 발생 빈도와 균형 맞는가?**
5. **메모리 룰과 충돌하는가?** → 그렇다면 작업 중단·시정 우선

> 본 원칙은 한 번의 세션 합의로 도출됨. 미래에 사용자가 다른 결정 (예: 자동 LLM 호출 허용) 을 하면 본 룰을 업데이트.
