# "What Didn't Work" 섹션 가이드

작업 내역 파일(`work/WXX-*.md`)에 시도했으나 실패한 접근법을 기록하는 선택적 섹션입니다. 종속 태스크가 있는 경우 후속 Worker의 불필요한 시행착오를 방지합니다.

**포함 시점:** 다음 조건 중 하나 이상에 해당하면 포함합니다:
- 최종 결과에 도달하기 전에 다른 접근법을 시도하여 실패한 경우
- 현재 태스크에 종속된 후속 태스크(Phase 2 등)가 존재하는 경우
- 직관적이지 않은 제약사항을 발견하여 후속 작업자에게 전달할 가치가 있는 경우

**기록 형식:**

```markdown
## What Didn't Work

| 접근법 | 실패 이유 | 대안 |
|--------|----------|------|
| [시도한 방법] | [실패/포기 이유] | [대신 선택한 방법 또는 권장 대안] |
```

**예시:**

```markdown
## What Didn't Work

| 접근법 | 실패 이유 | 대안 |
|--------|----------|------|
| regex로 AST 파싱 시도 | 중첩 구조에서 오탐 다수 발생 | @babel/parser 사용으로 전환 |
| 전체 파일 재작성 | 기존 포매팅/주석 손실 | Edit 도구로 부분 수정 |
| npm run build로 검증 | devDependencies 미설치 상태에서 실패 | npm ci 후 빌드로 변경 |
```

**종속 태스크 간 실패 정보 전달 원칙:**
- Phase N의 Worker가 기록한 "What Didn't Work"는 해당 작업 내역 파일에 저장됨
- Phase N+1의 종속 Worker는 계획서에서 종속성을 확인한 후, 선행 태스크의 작업 내역 파일을 읽어 실패 정보를 참고할 수 있음
- 오케스트레이터는 종속 태스크 호출 시 선행 작업 내역 경로를 별도로 전달하지 않음 (Worker가 계획서의 종속성 정보와 workDir 경로로 자율적으로 탐색)
- 실패 정보는 참고 사항이며, 후속 Worker가 다른 맥락에서 동일 접근법을 시도하는 것을 강제로 차단하지 않음

**작업 내역 파일 템플릿 내 위치:**

"What Didn't Work" 섹션은 작업 내역 파일의 다음 위치에 배치합니다 (선택적):

```markdown
# WXX. [작업명]

## 변경 파일
...

## 로드된 스킬              <-- 여기 (필수)
| 스킬명 | 매칭 방식 | 근거 |
|--------|----------|------|
| [스킬명] | skills 파라미터 / 명령어 기본 / 키워드 판단 / description 폴백 | [근거] |

## 수행 내용
...

## What Didn't Work          <-- 여기 (선택적)
| 접근법 | 실패 이유 | 대안 |
|--------|----------|------|
| ... | ... | ... |

## 판단 근거
...
```
