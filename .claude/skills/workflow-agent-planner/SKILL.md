# Plan

복잡한 작업을 분석하여 종속성/독립성을 파악하고, 병렬 실행 계획을 수립하는 스킬.

> 이 스킬은 workflow-orchestration 스킬이 관리하는 워크플로우의 한 단계입니다. 전체 워크플로우 구조는 workflow-orchestration 스킬을 참조하세요.

## 핵심 원칙

1. **명확성 우선**: 불명확한 요청은 명확해질 때까지 반복 질문
2. **종속성 분석**: 작업 간 의존관계 파악
3. **병렬화 극대화**: 독립적인 작업은 work 에이전트로 병렬 실행
4. **적응적 실행**: 간단한 작업은 즉시 실행, 복잡한 작업은 사용자 확인 후 실행
5. **PLAN 완전 명확화 필수**: WORK Step에서는 질문이 불가하므로 PLAN Step에서 100% 명확화
6. **Worker/Reporter 역할 분리**: Worker 태스크에 최종 보고서(report.md, summary 문서) 생성을 절대 포함하지 않음. 최종 보고서는 REPORT Step에서 reporter 에이전트가 전담. Worker 산출물은 작업 내역 파일(work/WXX-*.md)에 한정

## 터미널 출력 원칙

> 내부 분석/사고 과정을 터미널에 출력하지 않는다. 결과만 출력한다.

- **반환값**: 상태 1줄만 반환 (`작성완료`). 경로/메타정보/코드/목록/테이블 반환 금지 (common-reference.md 1줄 규칙 준수)
- **출력 금지**: 분석 과정, 판단 근거 설명, 요구사항 해석 과정, 작업 분해 과정 설명, "~를 살펴보겠습니다" 류, 중간 진행 보고
- 계획 수립 과정은 묵묵히 수행. 계획서 경로 출력 및 배너 호출은 오케스트레이터가 담당 (planner 에이전트가 직접 출력하지 않음)

## 오케스트레이션 연동 이해

planner가 계획서를 작성하기 전에 반드시 이해해야 할 오케스트레이션 컨텍스트입니다.

### 전체 흐름

```
PLAN Step → WORK Step(Phase 0 → Phase 1~N → Phase N+1) → REPORT Step → DONE Step
```

- **Phase 0**: `skill_mapper.py` 스크립트가 `skill-map.md` 생성 (필수, 스킵 불가)
- **Phase 1~N**: 계획서 기반 worker/explorer 실행
- **Phase N+1**: validator 실행 (implement/review 명령어만, research는 스킵)

### 오케스트레이터가 plan.md에서 추출하는 6개 필드

| 필드 | 설명 | 정확성 요구 |
|------|------|------------|
| `taskId` | 태스크 식별자 (W01, W02 등) | 고유하고 순차적이어야 함 |
| `phase` | 실행 Phase 번호 | 종속성과 일치해야 함 |
| `dependencies` | 선행 태스크 목록 | 존재하는 taskId만 참조 |
| `parallelism` | 병렬/순차 실행 여부 | Phase 내 동시 실행 가능성 반영 |
| `agentType` | 서브에이전트 타입 (worker-opus/sonnet, explorer) | 태스크 복잡도에 맞게 선택 |
| `skills` | 전문화 스킬 목록 | skill-catalog.md에 존재하는 스킬만 기재 |

> **이 6개 필드가 정확해야 오케스트레이터가 올바르게 디스패치할 수 있다.** 필드 누락이나 불일치 시 워커가 잘못된 설정으로 실행된다.

### 워커 실행 메커니즘

- 워커는 `plan.md`를 직접 읽어 자신의 taskId에 해당하는 작업 상세를 확인
- 종속 태스크 워커는 선행 작업 내역 파일(`work/WXX-*.md`)을 읽어 결과 확인
- **워커 간 직접 통신 불가** — 파일 시스템 기반 암묵적 공유만 가능

### 계획서 = Binding Contract

승인된 계획서는 변경 불가하다. planner 계획서 품질이 워커 실행 품질을 직결한다:

| 계획서 품질 문제 | 워커 실행 결과 |
|-----------------|---------------|
| 작업 상세가 모호 | 워커가 추가 질문 불가하므로 임의 판단 |
| 대상 파일이 부정확 | 워커가 잘못된 파일 수정 |
| 스킬 매핑이 부적절 | 워커가 비효율적으로 작업 |

## PLAN 가이드

planner 에이전트가 PLAN을 수행합니다.

### 충분한 사용자 티키타카 (핵심)

<!-- NOTE: 현재 planner(서브에이전트)는 AskUserQuestion을 직접 호출할 수 없습니다 (플랫폼 제약, GitHub Issue #12890).
     아래 기술된 반복 질의응답은 Tier 2에서 오케스트레이터 경유 방식으로 개선 예정입니다.
     현재는 planner가 추가 질문 없이 계획서를 작성하고, 오케스트레이터가 사용자 승인을 수행합니다. -->
- planner는 불명확한 부분을 계획서의 **가정 사항** 섹션에 명시하고, 오케스트레이터가 사용자 승인 시 확인한다
- 모호한 부분, 누락된 정보, 결정이 필요한 사항은 **가정 사항으로 기록**하여 추적 가능하게 한다
- WORK Step에서는 질문이 불가하므로 PLAN Step에서 최대한 명확화하되, 해소되지 않는 부분은 가정 사항으로 문서화
- 사용자가 "진행해", "OK" 등 승인 시에만 계획서 확정

### 명확성 검증 체크리스트

요청이 불명확하면 **불명확한 부분은 계획서의 가정 사항 섹션에 명시한다** (서브에이전트 제약으로 AskUserQuestion 직접 호출 불가). 다음 항목을 확인:

- 목표가 명확한가?
- 범위가 정의되어 있는가?
- 제약조건이 있는가?
- 예상 결과물이 명시되어 있는가?

**재질의 예시:**
```
"웹사이트 만들어줘"
-> "어떤 종류의 웹사이트인가요? (랜딩페이지, 대시보드, 블로그 등)"
-> "주요 기능은 무엇인가요?"
-> "사용할 기술 스택이 있나요?"
```

불명확한 부분은 계획서의 가정 사항 섹션에 명시하고, 오케스트레이터가 사용자 승인 과정에서 확인한다.

### 우선순위 규칙 (GR1~GR5 해소)

user_prompt.txt에 XML 태그가 포함된 경우, 아래 5가지 회색 영역(GR1~GR5)에 대해 우선순위 규칙을 적용한다.

| 회색 영역 | 규칙 | 우선순위 |
|----------|------|---------|
| GR1: 구현 방향 | 코드베이스 현실 > `<approach>` > planner 독자 판단 | 코드베이스 패턴을 먼저 확인하고, `<approach>` 명시 시 그에 따름 |
| GR2: 대상 파일 | `<context>` 명시 > planner 독자 탐색 | `<context>`에 파일이 기재된 경우 존재 확인(Read 1회)만 하고 재탐색 생략 |
| GR3: 기술 스택 | 코드베이스 기존 패턴 > `<constraints>` > planner 판단 | 기존 사용 기술을 우선 채택하고, `<constraints>`로 명시된 제약 준수 |
| GR4: 작업 범위 | `<scope>` 명시 > planner 확장/축소 | `<scope>` 태그가 있으면 계획서 In-Scope/Out-of-Scope의 기본값으로 사용 |
| GR5: 성공 기준 | `<criteria>` + planner 기술 기준 = 합집합 | `<criteria>` 항목을 계획서 완료 기준에 포함하고, planner 판단 기준을 추가로 기재 |

### prompt.txt 파싱 가이드

user_prompt.txt를 읽은 후, 계획 수립 전에 XML 태그 존재 여부를 확인하고 아래 규칙을 적용한다.

**(1) 태그 감지** — user_prompt.txt에서 `<goal>`, `<target>`, `<constraints>`, `<criteria>` 태그 존재 여부를 확인한다.

**(2) 태그 존재 시** — 각 태그 내용을 파싱하여 계획서 해당 섹션에 직접 매핑한다. `<context>` 태그에 명시된 파일은 신뢰 가능한 선행 정보이므로 직접 Read(재탐색 스킵)한다.

**(3) 태그 미존재 시(폴백)** — 기존 방식 유지. 전문을 자연어로 분석하여 요구사항을 추출한다.

**(4) 혼합 형식 허용** — 태그와 자연어가 혼재된 경우, 태그 내용을 우선하고 태그 외부 자연어는 보충 컨텍스트로 활용한다.

#### 8개 태그별 계획서 매핑 상세

| 태그 | 계획서 매핑 위치 | 활용 방법 |
|------|----------------|---------|
| `<goal>` | "작업 요약" 섹션 기반 | `<goal>` 내용을 작업 요약의 **목적** 서술에 반영. 단일 목표가 명확하므로 작업 목적 1문장으로 사용 |
| `<target>` | 코드베이스 탐색 시작점 | `<target>`에 명시된 파일/모듈을 탐색 우선 대상으로 설정. 경로가 정확하면 Glob 탐색 없이 직접 Read |
| `<constraints>` | 태스크 설계 시 제약 조건 | 워커 작업 상세의 "주의사항" 또는 "제약"으로 반영. 기술 스택 제약은 스킬 선택 시 참조 |
| `<criteria>` | 계획서 "완료 기준" 섹션 | `<criteria>` 항목을 완료 기준에 직접 포함. planner가 추가 기술 기준을 보완하여 합집합 구성 |
| `<context>` | 신뢰 가능 선행 정보 | cc:prompt가 탐색한 파일 경로는 존재 확인(Read 1회)만 하고 내용 재분석 생략. 핵심 패턴/함수명은 워커 작업 상세에 직접 활용 |
| `<approach>` | 구현 방향 기본값 | 태스크 작업 내용의 구현 방향으로 사용. 코드베이스 현실과 충돌 시 코드베이스 현실 우선(GR1 규칙) |
| `<scope>` | In-Scope/Out-of-Scope 기본값 | "작업 범위" 섹션의 초안으로 사용. `<scope>` 명시 내용을 In-Scope/Out-of-Scope에 그대로 적용 |
| `<reference>` | 태스크 작업 상세 참고 자료 | 워커 작업 상세의 "참조" 항목에 기재. 워커가 작업 시 해당 자료를 직접 Read/WebFetch할 수 있도록 경로/URL 명시 |

#### `<context>` 신뢰 원칙

cc:prompt 명령어가 코드베이스 탐색 또는 웹검색 후 기록한 `<context>` 내용은 **신뢰 가능한 선행 정보**로 취급한다.

- **파일 경로**: `탐색 파일` 항목에 기재된 경로는 파일 존재 여부만 확인(Read 1회)하고 내용 재분석은 생략
- **핵심 패턴/함수명**: `핵심 패턴`, `관련 함수/모듈` 항목은 재탐색 없이 워커 작업 상세에 직접 인용
- **외부 참조**: `외부 참조` 항목의 URL/버전 정보는 신뢰하고 워커 `참조` 항목에 그대로 전달
- **중복 탐색 방지**: `<context>`에 명시된 파일은 다시 Glob/Grep 탐색 대상에서 제외. 미명시 파일만 탐색 수행

### 코드베이스 탐색

요구사항 분석 후, 계획 수립 전에 반드시 코드베이스를 탐색하여 대상 파일과 현재 구현 패턴을 파악한다.

**탐색 절차:**

> `<context>` 태그에 명시된 파일은 이미 검증된 정보이므로 직접 Read한다. 미명시 파일만 Glob/Grep으로 탐색한다.

1. **Glob**으로 프로젝트 구조 파악 — 디렉터리 레이아웃, 관련 파일 위치 식별
2. **Grep**으로 요청 관련 키워드 검색 — 대상 파일 식별, 기존 구현 위치 확인
3. **Read**로 핵심 파일 확인 — 현재 구현 패턴, 코딩 컨벤션, 파일 라인 수 파악

**탐색 결과 활용:**

- 대상 파일의 **정확한 경로**와 **라인 수**를 계획서 워커 상세에 반영
- 현재 코드 패턴/컨벤션을 파악하여 워커 작업 지침에 반영
- 탐색 결과를 계획서 비고 > 현황 스냅샷에 정량 반영 (파일 수, 총 라인 수, 기존 패턴 등)

> **코드베이스 탐색 없이 계획서를 작성하면 대상 파일 경로가 부정확하거나, 현재 구현을 무시한 계획이 수립될 위험이 있다.**

### 작업 분해

요청을 개별 작업 단위로 분해:

```
예: "React 앱에 로그인 기능 추가"
-> T1: 로그인 UI 컴포넌트 생성
-> T2: 인증 API 연동
-> T3: 상태 관리 설정
-> T4: 라우트 가드 구현
-> T5: 테스트 작성
```

### 태스크 분배 검증 체크리스트

작업 분해 후 다음 항목을 반드시 검증:

- [ ] Worker 태스크에 `report.md` 생성이 포함되어 있지 않은가?
- [ ] Worker 태스크에 최종 보고서, summary 문서, 결과 리포트 등 보고서 성격의 산출물이 포함되어 있지 않은가?
- [ ] Worker의 산출물이 작업 내역 파일(`work/WXX-*.md`)로 한정되어 있는가?
- [ ] 탐색형 태스크(코드베이스 탐색, 웹 조사, 정보 수집)에 Explorer가 지정되어 있는가?
- [ ] 모든 Worker 태스크에 `스킬` 컬럼이 채워져 있는가?
- [ ] 프로젝트 스킬이 존재하면 모든 Worker 태스크에 적용했는가?

> **경고**: 최종 보고서 생성은 REPORT 단계에서 reporter 에이전트가 전담합니다. Worker에게 보고서 생성 태스크를 할당하면 역할 충돌이 발생합니다.

### 서브에이전트 타입 지정 가이드

작업 목록 테이블의 "서브에이전트" 컬럼에 각 태스크의 실행 에이전트를 지정합니다. 태스크의 성격과 복잡도에 따라 모델별 Worker 또는 Explorer를 선택합니다.

| 구분 | worker-opus | worker-sonnet | explorer |
|------|-------------|---------------|----------|
| 역할 | 복잡한 코드 생성/리팩토링, 아키텍처 변경 | 일반 구현/수정, 중간 복잡도 작업 | 코드베이스 탐색, 웹 조사, 정보 수집 |
| 모델 | opus | sonnet | sonnet |
| maxTurns | 50 | 50 | 30 |
| 적합 복잡도 | Tier 3 (> 15점) | Tier 2 (<= 15점) | - |
| 적합한 작업 예시 | 아키텍처 재설계, 다중 모듈 연동, 복잡한 리팩토링 | API 연동, 기능 구현, 버그 수정, 테스트 작성, 설정 파일 수정, 문서 갱신 | 코드 구조 분석, API 조사, 라이브러리 비교 |

**Worker 모델 선택 기준 (복잡도 티어 기반):**
- **Tier 3** (복잡도 > 15) -> **worker-opus**: 아키텍처 변경, 다중 모듈 연동, 복잡한 로직 구현
- **Tier 2** (복잡도 <= 15) -> **worker-sonnet**: 일반적인 기능 구현, 수정, 버그 수정, 설정 변경, 문서 수정

#### Opus 고려 트리거 조건

T2(10+) 태스크에서 아래 조건을 확인하여 worker-opus 배정을 검토합니다.

- [ ] T2(10점 이상) + 대상 파일 7개 이상 교차 수정 (다중 파일 변경 파급 효과 추적 필요)
- [ ] T2(10점 이상) + 아키텍처 키워드 포함: "경쟁 조건", "아키텍처 변경", "계층 재설계", "동기화", "하위 호환"
- [ ] T2(10점 이상) + 종합 판단 필요: "교차 검증", "이슈 분류", "트레이드오프 분석", "전략 수립"
- [ ] 선행 산출물 2개 이상을 통합 분석하는 메타 분석 태스크
- [ ] research 명령어의 최종 통합/전략 수립 태스크 (전체 연구 결과를 종합하는 Phase 최종 태스크)

**적용 규칙:**
- 트리거 조건 1개 해당: Opus 검토 권고 (planner 판단으로 Sonnet 유지 가능, 사유 기재)
- 트리거 조건 2개 이상 해당: Opus 강력 권고 (Sonnet 유지 시 명시적 사유 필수)

#### 명령어별 Opus 적합 패턴

명령어 맥락에 따라 Opus가 적합한 태스크 패턴을 우선 배정합니다.

| 명령어 | 태스크 패턴 | 권장 모델 |
|--------|-----------|---------|
| research | 최종 통합/전략 수립 (다수 선행 산출물 종합, 전략 비교표/로드맵 생성) | Opus 우선 |
| review | 아키텍처 리뷰 (계층 위반 분석, 크로스커팅 의존성 평가, 시스템 전체 영향도 검토) | Opus 우선 |
| implement | 경쟁 조건/동기화 (race condition 해소, TOCTOU 취약점 수정, 하위 호환 별칭 설계) | Opus 검토 |

#### 질적 모델 선택 4차원 체계

복잡도 점수 외에, 아래 4가지 차원의 질적 요소를 평가하여 Opus 배정 여부를 보완합니다.

| 차원 | 척도 | 설명 |
|------|------|------|
| 판단 난이도 | 0-3 | 0: 기계적 실행 / 1: 패턴 매칭 / 2: 종합 추론 / 3: 창의적 설계 |
| 창의성 요구도 | 0-2 | 0: 기존 패턴 반복 / 1: 일부 창의적 응용 / 2: 새 구조 설계 |
| 다중 파일 교차도 | 0-3 | 0: 단일 파일 / 1: 동일 모듈 내 2-3개 / 2: 모듈 간 4-6개 / 3: 7개 이상 교차 |
| 실패 위험도 | 0-2 | 0: 재실행 가능 / 1: 부분 복구 필요 / 2: 재실행 불가(보안·데이터 손실) |

**Opus 권장 규칙:** 4차원 중 3개 이상이 "중간 이상(판단 2+, 창의성 1+, 교차도 2+, 위험도 1+)"에 해당하면 Opus 권장.

**Worker vs Explorer 선택 기준:**
- 파일을 **수정/생성**해야 하면 -> **worker-opus/sonnet** (복잡도에 따라 선택)
- 정보를 **수집/분석**해야 하면 -> **explorer**
- 코드 + 웹 **통합 탐색**이 필요하면 -> **explorer**
- 동일 태스크에 탐색과 구현이 모두 필요하면 -> 탐색(explorer) + 구현(worker-*)으로 **분리**

### 스킬 선택 가이드 (3계층 모델)

Worker = 에이전트 스킬(자동) + 전문화 스킬(필수) + 프로젝트 스킬(반필수)

각 Worker 태스크에 최소 1개의 전문화 스킬을 작업 목록 테이블의 `스킬` 컬럼에 기재합니다.

**전문화 스킬 선택 절차:**

> **원칙: 태스크 성격이 명령어 기본 매핑보다 우선한다.** 명령어가 동일해도 태스크 내용의 성격에 따라 스킬이 달라진다.

0. **태스크 성격 분류 (최우선)** — 태스크 description을 읽고 아래 5분류 중 해당 유형을 결정한다

   | 태스크 성격 | 기본 전문화 스킬 | 판별 기준 |
   |------------|----------------|---------|
   | 구현/수정 | workflow-system-verification | 파일을 생성·수정·삭제하는 태스크 |
   | 검증/리뷰 | review-code-quality, review-architecture | 코드/문서의 품질·정합성을 검사하는 태스크 |
   | 디버깅 | debug-systematic | 버그 원인 분석, 에러 핸들링, 경쟁 조건 해소 |
   | 탐색/조사 | research-general, research-integrated | 정보 수집, 현황 분석, 비교 연구 |
   | 문서 | management-skill | 스킬 문서, 가이드, 템플릿 작성·수정 |

1. 명령어 기본 매핑 확인 (skill-catalog.md Command Default Mapping) — 0번에서 결정된 성격과 불일치하면 0번 우선
   - implement → review-code-quality, workflow-system-verification
   - review → review-requesting, review-code-quality
   - research → research-general, research-integrated
2. skill-catalog.md의 Skill Descriptions 섹션에서 각 스킬의 description 전문을 읽고, 태스크 성격에 가장 적합한 스킬을 판단 (Layer 2)
   - 각 스킬의 description에서 사용 목적, 적용 영역, 예시 트리거를 확인하여 태스크와 가장 부합하는 스킬을 선택한다
   - 키워드 매칭이 아니라 태스크의 **목적과 성격**이 스킬의 description과 일치하는지를 기준으로 판단한다
3. 최종 선택: 태스크에 가장 적합한 전문화 스킬 1-2개 기재

**태스크 성격별 스킬 매핑 테이블 (추가 참조):**

| 태스크 성격 | 추천 전문화 스킬 | 근거 |
|------------|-----------------|------|
| API 구현/연동 | framework-fastapi, framework-react | 프레임워크별 패턴 |
| 보안 검토 | review-security, debug-static-analysis | OWASP 체크리스트 |
| 성능 최적화 | review-performance | N+1 쿼리, 캐싱 |
| 테스트 작성 | testing-design, testing-coverage | ISTQB 기법 |
| DB 분석 | analyze-database | 스키마/인덱스 분석 |
| 웹 조사 | research-integrated, research-general | 통합 검색 |
| PR/코드 리뷰 | review-pr-integration, review-code-quality | 체크리스트 기반 |
| 프론트엔드 | review-frontend, design-frontend | React/Vue 패턴 |
| 아키텍처 | design-architect, design-patterns | UML/패턴 |

> **스킬 미지정 시 폴백:** 스킬 컬럼이 비어 있으면 워커가 자율적으로 판단하여 스킬을 선택한다. 그러나 자율 선택은 비효율적이므로 planner가 반드시 명시적으로 지정해야 한다.

**동일 패턴 태스크 일관성 참조:**

동일 명령어 + 유사 작업명의 이전 계획서가 있으면 스킬 배정 일관성을 확인한다. 최근 3건 동일 명령어 계획서 참조 권장.

- **목적**: 같은 유형의 태스크(예: API 연동, 문서 수정)에 일관된 스킬이 배정되도록 보장
- **확인 방법**: `.workflow/` 디렉터리에서 동일 명령어 계획서(plan.md)의 스킬 컬럼을 참조하여 배정 패턴 확인
- **적용 기준**: 이전 계획서의 스킬 배정이 현재 태스크 성격과 일치하면 동일하게 적용. 불일치하면 0번(태스크 성격 분류) 기준으로 재판단하고, 변경 사유를 비고에 기재

**프로젝트 스킬 적용:**

skill-catalog.md의 "Project Skills" 섹션을 확인합니다.
- 프로젝트 스킬이 존재하면: 모든 Worker 태스크에 `+ project-<도메인명>` 추가
- 프로젝트 스킬이 없으면: 전문화 스킬만 기재

### 종속성 분석

각 작업 간 관계 파악:

| 작업 | 선행 작업 | 병렬 가능 |
|------|----------|----------|
| T1 | 없음 | O |
| T2 | 없음 | O |
| T3 | 없음 | O |
| T4 | T2, T3 | X |
| T5 | T1, T2, T3, T4 | X |

**병렬 Phase 충돌 방지 가이드:**

병렬 Phase 내 동시 수정 파일 목록 교차 확인. 동일 파일 수정 시 순차 분리 또는 수정 영역 분리 명시.

- **확인 절차**: 병렬 배치 예정 태스크의 "대상 파일" 목록을 나열 후 중복 항목 검색
- **동일 파일 수정 시 처리**: (1) 순차 분리 — 한 태스크를 다음 Phase로 이동. (2) 수정 영역 분리 — 동일 파일 내 수정 섹션/함수가 겹치지 않으면 병렬 유지 가능. 겹치는 영역을 계획서 비고에 명시
- **경고**: 같은 Phase에서 같은 파일의 같은 영역을 수정하면 병렬 실행 시 충돌(편집 충돌, 덮어쓰기)이 발생할 수 있음

### 실행 계획 수립

**병렬 그룹 구성:**
```
Phase 1 (병렬): [T1, T2, T3] -> work 에이전트 3개 동시 실행
Phase 2 (순차): [T4] -> T2, T3 완료 후 실행
Phase 3 (순차): [T5] -> 모든 작업 완료 후 실행
```

### 태스크 복잡도 자동 평가

각 태스크의 복잡도를 수치화하여 분해 깊이와 리소스 배분을 결정합니다.

**복잡도 점수 산정 공식:**
```
복잡도 = action_verbs * 2 + connectors * 2 + domain + scope + risk + judgment * 2
```

| 요소 | 설명 | 점수 범위 |
|------|------|----------|
| `action_verbs` | 태스크 설명의 행위 동사 수 (생성, 수정, 삭제, 연동, 분석 등) | 0-5 (x2) |
| `connectors` | 접속사/조건 ("~하고", "~한 후", "~이면", 종속성, 공유 리소스) | 0-3 (x2) |
| `domain` | 도메인 전문성 필요도 (0: 일반, 1: 중간, 2: 전문) | 0-2 |
| `scope` | 영향 범위 (0: 단일 파일, 1: 모듈, 2: 시스템, 3: 크로스커팅) | 0-3 |
| `risk` | 부작용 위험도 (0: 안전, 1: 주의, 2: 높음) | 0-2 |
| `judgment` | 판단 난이도 (0: 기계적 실행, 1: 패턴 매칭, 2: 종합 추론, 3: 창의적 설계) | 0-3 (x2) |

> 전체 점수 범위: 0-29

**복잡도 티어:**

| 티어 | 점수 범위 | 분류 | 분해 깊이 가이드 |
|------|----------|------|----------------|
| Tier 1 | < 5 | 단순 (단일 파일 수정, 설정 변경) | 분해 불필요, 단일 태스크로 실행 |
| Tier 2 | 5 - 15 | 중간 (다중 파일 수정, 로직 변경) | 2단계 이하 분해 권장 |
| Tier 3 | > 15 | 복잡 (아키텍처 변경, 다중 모듈 연동) | 3단계 이상 분해 필수, 하위 태스크 분리 검토 |

**계획서 태스크 테이블에 복잡도 컬럼 추가:**

| ID | 작업 | 종속성 | Phase | 복잡도 |
|----|------|--------|-------|--------|
| W01 | 설정 파일 수정 | - | 1 | T1(2) |
| W02 | 인증 모듈 리팩토링 | W01 | 2 | T2(12) |
| W03 | API 게이트웨이 재설계 | W02 | 3 | T3(23) |

**평가 예시:**
- "settings.json에 필드 추가" -> 동사 1, 접속사 0, 도메인 0, 범위 0, 위험 0, 판단 0 = **T1(2)**
- "API 연동 후 에러 처리 및 재시도 로직 구현" -> 동사 3, 접속사 2, 도메인 1, 범위 1, 위험 1, 판단 1 = **T2(15)**
- "전체 Hook 시스템 리팩토링 및 하위 호환성 보장" -> 동사 2, 접속사 1, 도메인 2, 범위 3, 위험 2, 판단 2 = **T3(17)**
- "아키텍처 변경 + 경쟁 조건 해소 + 하위 호환 별칭 설계" -> 동사 3, 접속사 2, 도메인 2, 범위 3, 위험 2, 판단 3*2 = 6+4+2+3+2+6 = 23 = **T3(23)**

> **주의**: 복잡도 점수는 절대적 기준이 아닌 상대적 가이드라인이다. Tier 3 태스크가 하나의 Phase에 집중되지 않도록 분배를 검토한다.

### 워커 컨텍스트 예산 관리

각 워커(서브에이전트)는 독립적인 **200K 토큰 컨텍스트**를 보유한다. 태스크 설계 시 워커의 컨텍스트 소비를 예측하여 과부하를 방지해야 한다.

**워커 컨텍스트 소비 경로:**

| 소비 경로 | 예상 토큰 | 비고 |
|-----------|----------|------|
| plan.md 읽기 | ~5-10K | 전체 계획서 로드 |
| skill-map.md + 스킬 SKILL.md 로드 | ~10-30K | 스킬 수에 비례 |
| 선행 작업 내역 파일 읽기 | ~5-20K | 종속 태스크에서 발생 |
| 대상 파일 Read | ~파일 크기 비례 | 1000줄 ≈ 15-25K 토큰 |
| Bash 실행 결과 | ~가변 | 빌드/테스트 출력 |
| WebSearch/WebFetch 결과 | ~30-50K per call | 웹 조사 태스크 |

**태스크 크기 조절 가이드라인:**

| 항목 | 권장 | 분할 검토 기준 |
|------|------|---------------|
| 대상 파일 총 라인 | 500줄 이하 | 1000줄 초과 시 분할 |
| 작업 항목 수 | 3-5개 | 7개 초과 시 **분할 필수** |
| 탐색+구현 혼합 | 금지 | explorer + worker로 분리 |
| 웹 조사 (WebFetch) | 3회 이하 | 4회 이상 시 분할 권장 |

**워커 간 작업 항목 편차 제한:**

같은 Phase 내 워커들의 작업 항목 수 편차 최대 2. 초과 시 고항목 워커 분할 또는 저항목 워커 병합.

- **규칙**: 동일 Phase 내 워커별 작업 항목 수의 최대값과 최소값의 차이가 2를 초과하면 균형 조정 필수
- **고항목 워커 분할**: 항목 수가 많은 워커를 2개 워커로 분리하여 각 워커의 항목 수를 줄임
- **저항목 워커 병합**: 항목 수가 지나치게 적은 워커(1-2개)를 인접 워커와 합쳐 독립 Phase 구성 검토
- **예시**: Phase 2에 W02(7개), W03(2개)가 있으면 편차 5 → W02를 W02a(4개)/W02b(3개)로 분할하거나, W03 항목을 W02에 통합

**위험 신호 체크리스트** (태스크가 과대한 경우):

- [ ] 대상 파일이 5개 이상인가?
- [ ] 대상 파일 총 라인이 1000줄을 초과하는가?
- [ ] 작업 항목이 7개를 초과하는가?
- [ ] 탐색과 구현이 한 태스크에 혼합되어 있는가?
- [ ] 웹 조사가 4회 이상 필요한가?
- [ ] 단일 태스크에 WebSearch/Read 탐색과 Edit/Write 구현이 혼합되어 있는가?
- [ ] **무조건 분할 규칙**: 대상 파일 5개 이상 + 작업 항목 7개 초과가 동시에 해당되면 조건 수에 관계없이 즉시 분할 필수.

> **2개 이상 해당 시 태스크 분할 필수.** 분할하지 않으면 워커가 컨텍스트 한계에 도달하여 작업 품질이 저하된다.

### 계획서 작성

**계획서 템플릿:**

계획서 마크다운 템플릿은 별도 파일로 관리됩니다. 다음 절차로 사용합니다:

1. `.claude/skills/workflow-agent-planner/templates/plan.md`를 Read 도구로 로드

4. 작업 내용에 맞게 각 섹션을 작성

> 템플릿은 권장 구조이며, 내용에 따라 planner가 유연하게 조정할 수 있습니다.

### 자가 검증

계획서 작성 완료 후, 저장 전에 `reference/plan-quality-guide.md`의 체크리스트를 실행하여 품질을 검증한다.

**필수 검증 항목:**

- [ ] 모든 워커 태스크에 대상 파일 경로가 명시되어 있는가?

> 자가 검증에서 문제가 발견되면 저장 전에 수정한다.

**저장 위치:** `<workDir>/plan.md` (workDir = `.workflow/<YYYYMMDD-HHMMSS>/<workName>/<command>`)

### .context.json 업데이트 및 최종 컨펌

계획서 저장 후 .context.json 업데이트 및 최종 컨펌 절차는 오케스트레이터(workflow-orchestration)가 수행합니다. planner는 계획서 작성까지만 담당합니다.

오케스트레이터는 AskUserQuestion으로 다음 3가지 선택지를 제시합니다:

| 선택지 | 설명 |
|--------|------|
| **승인** | WORK 단계로 진행 |
| **수정** | prompt.txt에 피드백을 작성한 후 선택하면 planner가 계획을 수정하여 다시 컨펌 요청 |
| **중지** | 워크플로우를 중단 (CANCELLED) |

> **"수정" 동작:** 사용자가 prompt.txt에 수정 피드백을 작성한 후 "수정"을 선택하면, planner가 해당 피드백을 반영하여 계획을 재수립하고 다시 컨펌을 요청합니다. 피드백이 없으면 planner가 자체 판단으로 계획을 개선합니다.

## 병렬 실행 패턴

독립적인 작업들은 Task 도구의 work 에이전트를 사용하여 병렬 실행:

```
# 병렬 실행 예시 (단일 메시지에 여러 Task 호출)
Task(prompt="T1: 로그인 UI 컴포넌트 생성...")
Task(prompt="T2: 인증 API 연동...")
Task(prompt="T3: 상태 관리 설정...")
```

**중요:** 병렬 실행 시 반드시 단일 메시지에 여러 Task 호출을 포함해야 한다.

### 핵심 원칙

| 원칙 | 권장값 | 근거 |
|------|--------|------|
| Phase당 워커 수 상한 | **3-5개** (Anthropic 공식 문서) | "Start with 3-5 teammates for most workflows" (Agent Teams 공식 권장) |
| 컨텍스트 윈도우 제한 | **200K 토큰**/에이전트 (Anthropic 공식 문서) | 각 서브에이전트는 독립적인 200K 컨텍스트 윈도우 보유. 상세 결과 반환 시 메인 에이전트 컨텍스트 소비 급증에 주의 |
| 태스크 그룹화 | 팀메이트당 **5-6개** 태스크 (Anthropic 공식 문서) | "Having 5-6 tasks per teammate keeps everyone productive without excessive context switching" |
| 공식 하드리밋 | **없음** (Anthropic 공식 문서) | "There's no hard limit on the number of teammates, but practical constraints apply" |
| 커뮤니티 실용 상한 | **3-4개** 초과 시 생산성 저하 보고 | 3-4개 초과 시 조율 오버헤드 증가, 수확체감 발생 |

<!-- 공식 하드리밋 아님, 운영 권장값: Phase당 워커 수 3-5개는 Anthropic Agent Teams 공식 권장 + 커뮤니티 운영 경험(3-4개)을 종합한 수치 -->
<!-- 토큰 비용: N개 서브에이전트 병렬 실행 시 토큰 소비는 N배 선형 증가 (Anthropic 공식 문서) -->

#### Phase당 최소 2명 병렬 가이드

독립 태스크가 2개 이상 존재할 때 병렬 배치를 필수로 적용한다.

- **규칙**: 독립 태스크 2개 이상 존재 시 병렬 배치 필수. Phase당 워커 1명인 경우 독립성 미충족 이유 명시
- **판단 기준**: 태스크 간 종속성이 없고(선행 작업 불필요) 동시 수정 파일이 겹치지 않으면 병렬 배치 가능
- **예외 허용**: 단계적으로만 진행 가능한 작업(예: DB 마이그레이션 → 코드 수정 → 검증)은 순차 유지 가능. 사유를 계획서 비고에 기재

> 상세 가이드는 `reference/parallel-execution-guide.md`를 참조하세요.

## 워커별 작업 상세 작성 방법론

계획서의 "작업 목록" 테이블 아래에 "워커별 작업 상세" 섹션을 추가하여 각 워커가 수행할 구체적인 작업을 명시합니다.

**핵심 목표:** Worker가 추가 질문 없이 실행할 수 있는 수준의 구체적인 실행 지침 제공.

### (a) H3 서브섹션 형식 규격

워커별 작업 상세는 테이블 형식이 아닌 **H3 서브섹션 + 번호 매기기** 형식을 사용한다.

**채택 근거:**
- 테이블 형식은 셀 내 줄바꿈이 불가하여 복잡한 작업 내용을 기술하기 어려움
- H3 서브섹션은 HTML 주석으로 작성 가이드를 내장할 수 있음
- 대상 파일, 산출물을 별도 라인으로 분리하여 가독성 확보

**규격:**

```markdown
### WXX: [작업명]

- 대상 파일: `path/to/file`
- 산출물: `work/WXX-작업명.md`

**작업 내용:**

1. [행위 동사로 시작하는 구체적 작업 항목]
2. [행위 동사로 시작하는 구체적 작업 항목]
3. [행위 동사로 시작하는 구체적 작업 항목]
4. [행위 동사로 시작하는 구체적 작업 항목]
```

**필수 요소 3가지:**
1. **대상 파일 경로** - 워커가 수정/생성할 파일 (상대 경로, 백틱)
2. **산출물 경로** - 작업 내역 파일 (`work/WXX-<설명>.md`)
3. **번호 매기기 작업 내용** - 4-5개 항목 (최소 3개, 최대 7개)

### (b) 작업 항목 상세도 가이드라인

각 작업 항목은 **Worker가 추가 질문 없이 실행 가능한 수준의 구체성**을 갖춰야 한다.

**좋은 예 / 나쁜 예 대조표:**

| 나쁜 예 | 좋은 예 | 개선 포인트 |
|---------|---------|-----------|
| "파일 수정" | "`SKILL.md`의 252-282줄 '워커별 작업 상세 테이블 작성 가이드' 섹션을 확장 개편" | 대상 파일, 위치, 행위 구체화 |
| "기능 추가" | "`src/auth.ts`의 `login()` 함수 다음에 `refreshToken()` 함수 신규 생성" | 함수명, 위치, 행위 특정 |
| "테이블 업데이트" | "섹션 가이드 테이블에 '작업 요약'(순번 4, 선택) 행을 '사용자 질의응답 내역' 행 다음에 추가" | 추가 위치, 내용 값 명시 |
| "설정 변경" | "`tsconfig.json`의 `compilerOptions.strict`를 `true`로 변경" | 파일, 키, 값 특정 |
| "리팩토링 수행" | "`utils.ts`의 `formatDate()` 함수 3곳 중복 호출을 `useDateFormatter` 훅으로 추출" | 대상, 횟수, 추출 방식 특정 |

**상세도 자가 검증 질문:** "이 항목만 읽고 Worker가 어떤 파일의 어떤 위치에서 무엇을 어떻게 변경해야 하는지 알 수 있는가?"

### (c) 대상 파일 명시 규칙

| 규칙 | 설명 | 예시 |
|------|------|------|
| 상대 경로 | 프로젝트 루트 기준 상대 경로 사용 | `src/auth/login.ts` |
| 백틱 필수 | 모든 파일 경로를 백틱으로 감싸기 | `` `path/to/file` `` |
| 링크 형식 | 기존 파일은 `` [`경로`](경로) `` 링크 적용 권장. 미존재 파일(신규 생성 예정, 산출물 경로)은 백틱만 유지 | `` [`src/auth.ts`](src/auth.ts) `` |
| 단일 파일 | 대상 파일 1개인 경우 | `- 대상 파일: \`src/auth.ts\`` |
| 다중 파일 | 주 대상과 부 대상을 구분 | `- 대상 파일: \`src/auth.ts\`` (주), `\`src/types.ts\`` (부) |
| 디렉터리 | 디렉터리 전체가 대상일 때 | `- 대상 파일: \`src/components/\`` (디렉터리 내 전체) |

**다중 파일 표기:**
```markdown
- 대상 파일: `src/auth.ts` (주), `src/types.ts`, `src/utils.ts` (부)
```

> **이중 접두사 버그 주의**: 링크 경로 생성 시 `.workflow/` 등의 접두사가 중복 결합되지 않도록 주의한다. 예: `[../.workflow/../.workflow/path](...)` 형태는 버그이므로 경로를 단일 기준점에서 한 번만 결합해야 한다.

### (d) 작업 내용 번호 매기기 원칙

| 원칙 | 설명 | 위반 예 | 준수 예 |
|------|------|---------|---------|
| 행위 동사 시작 | 각 항목은 구체적 행위 동사로 시작 | "인증 로직에 대해서" | "**추가**: 인증 로직을 `authMiddleware`에 삽입" |
| 단일 행위 | 한 항목에 복수 행위 병합 금지 | "X 수정 및 Y 추가" | (1) X 수정 (2) Y 추가 (별도 항목) |
| 위치 특정 | 파일 내 구체적 위치/섹션/함수명 명시 | "함수 추가" | "`login()` 함수 다음에 `logout()` 함수 추가" |
| 무엇+어떻게 | 변경의 "무엇"과 "어떻게"를 모두 기술 | "섹션 추가" | "'작업 요약' H2 섹션을 '사용자 질의응답 내역' 섹션 다음에 추가" |

**허용 행위 동사 목록:** 생성, 수정, 추가, 삭제, 이동, 분리, 추출, 병합, 분석, 검증, 갱신, 확장, 개편, 변환

**작성 예시:**

```markdown
### W01: 인증 모듈 구현

- 대상 파일: `src/auth/login.ts` (주), `src/auth/types.ts` (부)
- 산출물: `work/W01-인증-모듈.md`

**작업 내용:**

1. `src/auth/types.ts`에 `LoginRequest`, `LoginResponse` 인터페이스 신규 생성
2. `src/auth/login.ts`에 `loginWithCredentials()` 비동기 함수 추가 (fetch API 사용)
3. `loginWithCredentials()` 내부에 HTTP 401/403 에러 분기 처리 추가
4. `src/auth/index.ts`에서 신규 함수 re-export 구문 추가
```

### (e) 선행 결과 참조 명시 규칙

종속 태스크(선행 태스크에 의존하는 태스크)는 선행 산출물을 읽어야 한다. 이때 워커가 선행 파일을 정확히 찾고 읽을 수 있도록 **산출물 경로 + 읽어야 할 섹션명**을 명시해야 한다.

**필수 명시:**

각 종속 태스크 문단의 시작(대상 파일 정보 다음)에 선행 산출물 참조를 다음 규격으로 명시:

```markdown
- 참조: `work/WXX-작업명.md` > ## 섹션명
- 참조: `work/WYY-작업명.md` > ## 섹션명
```

**참조 규칙:**

| 항목 | 규격 | 예시 |
|------|------|------|
| 경로 | 상대 경로, work 디렉터리 기준 | `work/W01-모델선택.md` |
| 백틱 필수 | 경로를 백틱으로 감싸기 | `` `work/W01-모델선택.md` `` |
| 섹션명 | 선행 파일에 존재하는 H2 섹션명 | `## 핵심 발견` |
| 다중 참조 | 여러 선행 산출물이 필요하면 각각 별도 라인 | (1) W01 참조 (2) W02 참조 |
| 대상 파일 다음 | 참조 항목은 "대상 파일" 정보 다음에 기재 | 대상 파일 → 산출물 → 참조 순서 |

**참조 예시:**

```markdown
### W04: P1 모델선택 체계 확장 (P1-1, P1-3, P1-8)

- 대상 파일: `path/to/SKILL.md`
- 산출물: `work/W04-P1-모델선택-체계.md`
- 참조: `work/W01-P0-모델선택-작업분할.md` > ## 핵심 발견
- 참조: `work/W02-P0-스킬선택.md` > ## 핵심 발견

**작업 내용:**

1. ...
```

**무엇이 들어가는가?**

선행 파일이 10KB, 선행 섹션이 3KB일 때, 섹션명 명시는 워커가 전체 파일을 로드하는 대신 해당 섹션만 구간 읽기(offset + limit)로 가능하게 하여 **컨텍스트 30% 절감**을 기대할 수 있다.

## "작업 요약" 섹션 작성 기준

"작업 요약"은 메타 정보의 "요청"과 명확히 구분되는 별도의 역할을 가진다.

**"요청" vs "작업 요약" 차별화:**

| 구분 | 요청 (메타 정보) | 작업 요약 |
|------|----------------|----------|
| 작성자 | 사용자 | planner |
| 성격 | 사용자 원문 그대로 기록 | planner가 분석/해석한 작업 개요 |
| 내용 | "로그인 기능 추가해주세요" | "OAuth 2.0 기반 인증 모듈을 3개 파일에 걸쳐 구현하고, 기존 세션 관리와 통합한다" |
| 포함 요소 | 원문 | 목적 + 접근 방식 + 예상 산출물 |

**작성 원칙:**
1. 1-3문장으로 작업 전체의 **목적**, **접근 방식**, **예상 산출물**을 요약
2. 메타 정보 "요청"의 단순 반복이 아닌, planner의 분석 결과를 반영
3. 복잡도 T2+ 태스크가 있거나 워커 2명(W02) 이상인 경우 권장
4. 단순 T1 단일 워커 계획에서는 생략 (메타 정보 "요청"으로 충분)

## "작업 범위" 섹션 작성 기준

작업 범위는 In-Scope/Out-of-Scope를 명시적으로 분리하여 범위 크리프를 방지한다.

**command별 권장 조건:**

| command | 권장 수준 | 근거 |
|---------|----------|------|
| research | 권장 | 연구 대상과 제외 대상의 명확한 경계 필요. 기존 비고 내 "연구 범위 경계" 서브섹션을 대체 |
| architect | 권장 | 설계 범위와 비설계 범위의 구분이 설계 품질에 직결 |
| analyze | 권장 | 분석 대상의 범위 명확화가 분석 품질에 직결 |
| implement | 복잡 시 | 수정 대상/비대상 파일 구분이 필요한 복잡한 구현에서 활용 |
| refactor, review, build, framework | 해당 없음 | 범위가 자명하거나 다른 섹션(비고)에서 충분히 다뤄짐 |

**In-Scope / Out-of-Scope 작성 원칙:**
- 포함 항목은 구체적 행위 또는 대상으로 기술 (예: "OAuth 2.0 인증 모듈 구현", "기존 세션 관리 통합")
- 제외 항목은 혼동 가능한 인접 범위를 명시적으로 배제 (예: "소셜 로그인(Google/GitHub) 연동은 제외", "UI 디자인 변경은 제외")
- 범위가 자명한 단순 작업(T1 단일 워커)에서는 생략
