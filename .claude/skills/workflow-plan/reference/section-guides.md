# 섹션 작성 가이드

## 선택적 섹션 판단 기준

_guide.md에서 상세 가이드를 제공하며, 아래는 planner가 즉시 참조할 수 있는 요약이다.

| 섹션 | 포함 조건 | 생략 조건 |
|------|----------|----------|
| 작업 요약 | T2+ 태스크가 있거나 W02 이상의 다중 워커 (선택) | T1 태스크 1개만 있는 단순 계획, 또는 메타 정보 "요청"으로 충분히 이해 가능한 경우 |
| 작업 범위 | research/architect/analyze 권장, 복잡 implement 가능 (선택) | 범위가 자명한 단순 작업 |
| 완료 기준 | 대부분의 계획서 (권장) | 단일 T1 태스크만 있는 매우 단순한 계획 |
| 리스크 평가 | T2 이상 태스크가 1개라도 있을 때 (권장) | T1 태스크만 구성 |
| 가정 사항 | 불명확한 가정이 존재할 때 | 모든 요구사항이 100% 명확화된 경우 |
| 비고 | 대부분의 계획서 (권장) | 추가 정보가 전혀 없는 경우 |
| 워커별 작업 상세 | 대부분의 계획서 (권장) | 워커 1명(W01만)이고 작업 내용이 단순 |
| 작업 흐름도 | 대부분의 계획서 (권장) | 워커 1명이거나 종속성 없는 단순 병렬 |

**핵심 원칙**: 불필요한 섹션을 형식적으로 채우는 것보다, 정말 필요한 섹션에 풍부한 정보를 제공하는 것이 우선이다.

## 비고 섹션 작성 가이드

비고 섹션은 Worker가 작업 시작 전 현재 상황을 객관적으로 파악하는 데 핵심적인 역할을 한다. command별로 다음 정량적 데이터를 반드시 포함한다.

**command별 필수 정량 데이터:**

| command | 필수 정량 데이터 유형 | 예시 |
|---------|---------------------|------|
| implement | 대상 파일 수, 수정 예상 라인 규모, 현행 동작 요약(수치) | "대상 파일 3개, 총 450줄, 현행: 함수 12개" |
| research | 현황 스냅샷(크기, 파일 수, 기간 등), 연구 범위 경계 bullet | "총 2.3MB, 301개 파일, 36개 디렉터리, 기간 2일" |
| review | 검토 대상 파일 수, 코드 라인 규모, 검토 범위 정량화 | "대상 15개 파일, 총 2,400줄, 함수 45개" |
| refactor | 리팩토링 전 상태 수치(코드 라인, 함수 수, 중복도 등) | "현재 800줄, 함수 20개, 중복 코드 3건" |
| architect | 현행 구조 요약(모듈 수, 의존성 수, 레이어 수 등) | "모듈 8개, 의존성 15개, 3-tier 구조" |

**비고 서브섹션 구조화 원칙:**
- 비고 내용은 반드시 서브섹션(### 레벨)으로 구분
- 최소 1개의 정량적 현황 서브섹션 포함 (예: "### 현황 스냅샷", "### 현황 분석")
- 서브섹션 내에 테이블 또는 bullet으로 수치 데이터 제시
