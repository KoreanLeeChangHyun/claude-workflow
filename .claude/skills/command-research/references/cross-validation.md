# 다중 소스 교차 검증 워크플로우

research 스킬 사용 시 수집된 정보의 신뢰성을 높이기 위한 교차 검증 가이드입니다.

## 최소 3개 소스 수집 원칙

모든 핵심 주장(claim)에 대해 최소 3개의 독립 소스에서 확인합니다.

| 소스 유형 | 예시 | 독립성 판단 기준 |
|----------|------|----------------|
| 공식 문서 | 프레임워크 공식 사이트, RFC, 표준 문서 | 원본 출처이므로 항상 독립 |
| 학술/기술 논문 | arXiv, IEEE, ACM | 다른 저자의 논문이면 독립 |
| 기술 블로그 | 개인/기업 블로그 | 동일 저자가 아닌 경우 독립 |
| 커뮤니티 | Stack Overflow, Reddit, GitHub Issues | 동일 스레드 내 답변은 비독립 |

**비독립 소스 주의**: 동일 원본에서 파생된 자료(예: A 블로그가 B 블로그를 인용)는 독립 소스로 카운트하지 않습니다.

## 소스 간 모순 탐지 체크리스트

정보 수집 후 다음 항목을 점검하여 소스 간 모순을 탐지합니다.

- [ ] **수치/날짜 불일치**: 소스별 수치(성능 벤치마크, 릴리스 날짜 등)가 일치하는가?
- [ ] **인과 관계 모순**: 한 소스의 "원인 A -> 결과 B" 주장이 다른 소스와 상충하는가?
- [ ] **범위/맥락 차이**: 소스별로 동일 용어를 다른 의미로 사용하고 있지 않은가?
- [ ] **시점 차이**: 정보의 작성 시점이 달라 버전/상황 차이가 발생하는가?
- [ ] **이해관계 편향**: 특정 소스가 자사 제품/기술을 옹호하는 편향이 있는가?

## 모순 발견 시 처리 절차

1. **모순 기록**: 어떤 소스에서 어떤 주장이 충돌하는지 명확히 기록
2. **원본 추적**: 각 주장의 원본 출처(primary source)를 추적하여 파생 오류 가능성 제거
3. **시점 확인**: 최신 소스의 정보를 우선하되, 시점 차이를 명시
4. **리포트 반영**: 모순이 해소되지 않는 경우 리포트에 양측 주장을 병기하고 판단 근거 제시

## 교차 검증 결과 기록 템플릿

리포트의 "조사 결과" 섹션에서 핵심 주장별로 다음 형식을 사용합니다.

```markdown
### [핵심 주장/사실]

| 소스 | 내용 | 일치 여부 |
|------|------|----------|
| [소스1 이름](URL) | [해당 소스의 관련 내용 요약] | 일치 / 부분 일치 / 불일치 |
| [소스2 이름](URL) | [해당 소스의 관련 내용 요약] | 일치 / 부분 일치 / 불일치 |
| [소스3 이름](URL) | [해당 소스의 관련 내용 요약] | 일치 / 부분 일치 / 불일치 |

**검증 결과**: [검증 판정 - 확인됨 / 부분 확인 / 미확인 / 모순 발견]
**비고**: [모순 발견 시 상세 설명, 또는 추가 조사 필요 사항]
```

## 검증 수준 가이드

| 주장 중요도 | 최소 소스 수 | 검증 수준 |
|------------|------------|----------|
| 핵심 결론에 직접 영향 | 3개 이상 | 전체 교차 검증 (템플릿 사용) |
| 보조 정보/배경 지식 | 2개 이상 | 간이 검증 (소스 간 일치 확인) |
| 일반 상식/널리 알려진 사실 | 1개 | 출처 명시만 |
