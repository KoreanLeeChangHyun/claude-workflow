# AT-11: Verification Integrity

## Evaluation Dimension

Assess whether the work logs (work_md_aggregated) contain actual test or
execution results — i.e., whether the worker ran tests, commands, or checks
and cited real output rather than asserting success without evidence.

## Scoring Criteria

| Score | Verdict | Criterion |
|-------|---------|-----------|
| 5     | PASS    | Work logs contain ≥ 1 explicit test run or command output (e.g., pytest output, git status, file listing) per major task |
| 4     | PASS    | ≥ 80% of tasks have at least one execution result citation; minor gaps in low-risk tasks |
| 3     | WARN    | 50–79% of tasks cite execution results; some tasks assert completion without showing output |
| 2     | FAIL    | < 50% of tasks have execution evidence; most completion claims lack supporting output |
| 1     | FAIL    | Work logs contain no execution or test results; all completion claims are assertions only |

## Bias Mitigation

Distinguish between tasks where execution evidence is expected (code changes,
test runs) vs. documentation tasks where a file-listing or diff suffices.

---

## Prompt

You are an impartial workflow quality judge evaluating a single audit item.

**Item:** AT-11 — Verification Integrity

**Evaluation focus:**
Did the worker actually run tests or verification commands, and are the results
cited in the work logs?

Look for:
- Test framework output (pytest, unittest results)
- `git status` or `git diff` output showing file changes
- Command output (ls, file contents verification)
- Explicit error/success output from tools or scripts

Assertions like "tests pass" or "implementation complete" without cited output
do NOT count as verification evidence.
