# AT-12: Migration Regression Safety

## Evaluation Dimension

Assess whether the work explicitly addresses backward compatibility, regression
risk, and rollback strategy — particularly for schema migrations, file renames,
API changes, or any modification affecting existing callers.

## Scoring Criteria

| Score | Verdict | Criterion |
|-------|---------|-----------|
| 5     | PASS    | Work explicitly states backward-compatibility guarantees AND documents rollback procedure AND identifies breaking-change risks |
| 4     | PASS    | Work addresses backward compatibility and at least one of: rollback procedure or breaking-change risk identification |
| 3     | WARN    | Work mentions compatibility or regression risk but without concrete rollback plan or specific affected callers |
| 2     | FAIL    | Work modifies existing interfaces without any discussion of backward compatibility or regression risk |
| 1     | FAIL    | Work introduces migration or interface change with no backward compatibility, rollback, or regression analysis |

**Note (implement/refactor/build only):** This item applies only to
`implement`, `refactor`, and `build` commands. For `research` and `review`
commands, this item is skipped (score=None, verdict=PASS,
evidence="skipped: not applicable").

## Bias Mitigation

Apply only when the work involves schema changes, renames, API modifications,
or removals of existing fields/endpoints; do not penalize for purely additive
changes that introduce no backward compatibility risk.
